# Project Control Prompt Package Specification

## Status

Draft

## Purpose

This document defines the Prompt Package model for Project Control Gateway.

A Prompt Package is a generated execution contract for Codex or another AI executor.

Its purpose is to give the AI agent a narrow, explicit and validated task context instead of asking it to infer work from many Markdown files or chat history.

Core principle:

```text id="wop9mh"
AI must not invent execution context.
Python builds the prompt package from approved project state.
Codex executes only the generated package for the selected Task.
```

## Problem

AI-assisted execution becomes unstable when Codex receives broad or ambiguous context.

Common risks:

```text id="fzj4wl"
- Codex reads too many files and invents scope;
- Codex treats Initiative or Epic as executable work;
- Codex edits files outside expected boundaries;
- acceptance criteria are implicit or missing;
- verification expectations are unclear;
- Human Owner approval is lost in chat history;
- task state and prompt text drift apart.
```

Prompt Package solves this by generating execution instructions from machine-readable Task state.

---

# 1. Definition

A Prompt Package is a generated, task-specific instruction document.

It is built from:

```text id="ul9j2c"
AI_PROJECT/state/tasks.json
AI_PROJECT/state/plan.json
```

and rendered into:

```text id="r69f7l"
AI_PROJECT/generated/CODEX_PROMPT.md
```

The package is generated by:

```bash id="kbovrb"
python scripts/taskctl.py prompt build --write
```

or printed to stdout by:

```bash id="4mv7gk"
python scripts/taskctl.py prompt build
```

## Prompt Package Is Not Source Of Truth

Prompt Package is derived output.

It must not be manually edited to change project state.

If Prompt Package is wrong, the underlying Task state must be corrected through `taskctl.py`, then the prompt must be rebuilt.

# 3. Current Implementation

Current CLI:

```bash id="55p5jr"
python scripts/taskctl.py prompt build
```

Supported options:

```text id="xuo71y"
--task <TASK_ID>       Build prompt for a specific Task.
--write                Write prompt to AI_PROJECT/generated/CODEX_PROMPT.md.
--out <PATH>           Write prompt to custom output path.
--allow-inactive       Allow prompt build for non-executable statuses.
--skip-plan-check      Validate tasks without checking plan references.
```

Default behavior:

```text id="d56ig6"
If --task is not provided, taskctl.py uses current_task_id.
If no current task exists, prompt build fails.
If task status is not executable and --allow-inactive is not provided, prompt build fails.
```

# 6. Prompt Package Structure

A Codex Prompt Package should contain these sections:

```text id="pihayx"
1. System Header
2. Active Control Fields
3. Repository and Task Identity
4. Parent Planning Context
5. Task Context
6. Details
7. Scope
8. Out of Scope
9. Allowed Files
10. Acceptance Criteria
11. Review Instructions
12. Execution Rules
13. Suggested Lifecycle Commands
14. Final Report Requirements
15. Stop Conditions
```

The package should be short enough to fit into the executor context without requiring the AI to read the whole project state.

---

# 7. Section Requirements

## 7.1 System Header

The prompt must start with:

```text id="kpvh0u"
[SYSTEM]
```

This marks the package as an execution instruction block.

## 7.2 Active Control Fields

Prompt Package must include:

```text id="0kj8oj"
Active Role
Active Stage
Active Document
Expected Result
```

These are derived from Task fields:

```text id="gujs9t"
task.active_role
task.active_stage
task.active_document
task.expected_result
```

Default values may be used only when fields are empty.

Suggested defaults:

```text id="a391c0"
Active Role: Codex Executor
Active Stage: Task Execution
Active Document: AI_PROJECT/generated/CODEX_CURRENT.md
Expected Result: Task completed according to acceptance criteria
```

## 7.3 Repository and Task Identity

Prompt Package must include:

```text id="gt9k7j"
Repository
Task ID
Task Title
Task Status
Verification Mode
```

These fields allow Codex to identify the exact executable unit.

## 7.4 Parent Planning Context

Prompt Package may include:

```text id="5o81m6"
Initiative: INIT-001 — ...
Epic: EPIC-001 — ...
```

This context is informational only.

Codex must not expand work based on Initiative or Epic title.

## 7.5 Task Context

Prompt Package must include Task summary.

If summary is missing, the package should explicitly say:

```text id="mfgisw"
No summary provided.
```

Codex must not infer missing context silently.

## 7.6 Details

If Task description exists, Prompt Package should include it under `Details`.

If description is missing, this section may be omitted.

## 7.7 Scope

Prompt Package must include Task scope.

Scope defines what Codex is allowed to do.

If scope is missing, the package must warn:

```text id="mj364y"
No explicit scope defined. Ask Human Owner before making broad changes.
```

## 7.8 Out of Scope

Prompt Package must include out-of-scope items.

If out-of-scope is missing, the package should use conservative default:

```text id="4dqhio"
Anything not required by the task and acceptance criteria.
```

## 7.9 Allowed Files

Prompt Package must include allowed files.

Allowed files define the file boundary for execution.

If allowed files are missing, the package must warn:

```text id="mfcc28"
Not specified. Do not edit files until allowed files are clarified.
```

## 7.10 Acceptance Criteria

Prompt Package must include acceptance criteria.

If acceptance criteria are missing, the package should provide minimal fallback only for validation safety:

```text id="m3cfcq"
- Task is completed without breaking validation.
- Generated files are updated only through CLI/render commands.
```

Recommended future policy:

```text id="zk2cno"
A Task should not be executable without explicit acceptance criteria.
```

## 7.11 Review Instructions

If Task has review instructions, Prompt Package must include them.

If no review instructions exist, the section may be omitted.

## 7.12 Execution Rules

Prompt Package must include execution rules.

Minimum rules:

```text id="7a126m"
- Do not edit AI_PROJECT/state/*.json manually.
- Do not edit AI_PROJECT/events/*.jsonl manually.
- Do not edit AI_PROJECT/generated/*.md manually unless explicitly instructed; prefer CLI render/build commands.
- Stay within Allowed Files and Scope.
- If task state must change, report the required taskctl command instead of editing state by hand.
- At the end, report changed files, checks run, result, and unresolved risks.
```

## 7.13 Suggested Lifecycle Commands

Prompt Package may include useful lifecycle commands.

Example:

```bash id="8fym1m"
python scripts/taskctl.py task transition TASK-001 --to in_progress
python scripts/taskctl.py task transition TASK-001 --to in_review
python scripts/taskctl.py validate
```

These commands are suggestions, not permission to bypass owner gates.

## 7.14 Final Report Requirements

Prompt Package should require Codex to report:

```text id="5af40m"
- changed files;
- commands run;
- validation result;
- generated files updated;
- acceptance criteria status;
- unresolved risks;
- owner action required.
```

## 7.15 Stop Conditions

Prompt Package must define stop conditions.

Codex must stop if:

```text id="mt4bbg"
- scope is unclear;
- allowed files are missing or too broad;
- required parent Epic is missing;
- task status is not executable;
- requested operation needs unsupported command;
- validation fails;
- work would change project control rules without approved Evolution Change;
- Human Owner approval is required but missing.
```

```text id="y86m0r"

in_review

```

Recommended stricter future policy:

```text id="10vi45"

changes_requested
```

`--allow-inactive` may be used for inspection, debugging or review, but not as normal execution flow.

# 9. Current Task Rule

If `--task` is not provided, Prompt Package builder should use:

```text id="4e278i"
AI_PROJECT/state/tasks.json.current_task_id
```

If no current task is selected, prompt build must fail with an explicit error.

Expected error:

```text id="ti6f4e"
NO_TASK_SELECTED
```

Current task must be set through:

```bash id="gj5ebr"
python scripts/taskctl.py current set TASK-001
```

Codex must not set current task by editing `tasks.json`.

# 10. Prompt Build Events

When Prompt Package is written to file, the system should append an audit event.

Event command:

```text id="tisy32"
prompt.build
```

Entity type:

```text id="jtcu2d"
task
```

Entity ID:

```text id="kenqb8"
TASK-001
```

Example payload:

```json id="5re0a5"
{
  "out": "AI_PROJECT/generated/CODEX_PROMPT.md"
}
```

Prompt build does not change semantic task state.

Therefore `revision_before` and `revision_after` may be equal.

---

# 11. Manual Edit Policy

Generated Prompt Package is protected generated output.

Forbidden:

```text id="dier8h"
- manual edits to CODEX_PROMPT.md to change scope;
- manual edits to CODEX_PROMPT.md to add allowed files;
- manual edits to CODEX_PROMPT.md to change acceptance criteria;
- manual edits to CODEX_PROMPT.md to override lifecycle state;
- treating CODEX_PROMPT.md as source of truth.
```

Allowed:

```text id="z31n04"
- read CODEX_PROMPT.md;
- delete and regenerate CODEX_PROMPT.md;
- write CODEX_PROMPT.md through taskctl.py prompt build --write;
- write custom prompt output through taskctl.py prompt build --out <PATH>.
```

# 14. Context Budget Rules

Prompt Package must be concise.

Recommended limits:

```text id="afzhq6"
Prompt Package: up to 300 lines
Scope: up to 30 items
Allowed Files: up to 50 items
Acceptance Criteria: up to 30 items
Recent events: omitted in MVP
Full audit log: never included by default
Full tasks list: never included by default
```

Prompt Package should not include:

```text id="jwpb18"
- full tasks.json;
- full plan.json;
- full audit logs;
- unrelated tasks;
- unrelated epics;
- long documentation dumps;
- old chat history.
```

If more context is needed, Codex should request a targeted file or command.

## Context Pack Boundary

When Codex needs additional documentation context, use `contextctl.py` to generate a bounded Context Pack:

```bash
python scripts/contextctl.py pack build --task <TASK_ID> --write
```

Context Pack output is derived retrieval context. It may help Codex decide which source sections to inspect, but it must not change the Prompt Package contract.

Context Pack must not:

```text
- expand Task scope;
- add allowed files;
- add acceptance criteria;
- override out-of-scope items;
- replace source documents or Task state;
- include full tasks.json, full docs.json or full audit logs by default.
```

The default retrieval policy excludes generated files, inactive documents, archived documents, deprecated documents, templates and examples unless explicitly allowed by a `contextctl.py` include flag.

# 15. Relationship To Generated Current Task

`CODEX_CURRENT.md` is a readable current-task view.

`CODEX_PROMPT.md` is an execution package.

Difference:

```text id="b67t69"
CODEX_CURRENT.md:
  readable task dashboard

CODEX_PROMPT.md:
  direct instruction package for Codex execution
```

Both are generated from Task state.

Neither is source of truth.

---

# 16. Relationship To planctl.py

Prompt Package may include Initiative and Epic titles from `plan.json`.

But `planctl.py` remains responsible only for upper-level plan state.

Prompt Package must not mutate plan state.

If planning state must change, Codex must use:

```bash id="p3eo45"
python scripts/planctl.py ...
```

not manual edits.

---

# 17. Relationship To taskctl.py

Prompt Package is built by `taskctl.py`.

`taskctl.py` owns:

```text id="d2esmn"
Task state
Current Task
Task generated Markdown
Codex Prompt Package
Task audit events
```

Prompt Package build must not bypass task validation.

Before building the package, task state must be valid.

`contextctl.py` may read Task state to derive a search query for a Context Pack, but it does not mutate Task state and does not make retrieved context executable scope.

---

# 18. Relationship To evolutionctl.py

If a task changes AI Development System rules, lifecycle, schemas, command catalog, prompt behavior or protected files policy, it must be linked to an approved Evolution Change in future `evolutionctl.py`.

Prompt Package for such work should include:

```text id="4qpdae"
Change ID
Change status
Affected areas
Migration impact
Owner approval
```

Until `evolutionctl.py` exists, such requirements should be represented in Task fields:

```text id="odmb91"
scope
out_of_scope
review_instructions
notes
acceptance_criteria
```

---

# 19. Validation Rules

Prompt Package build must validate:

```text id="dlfbj4"
- tasks.json exists;
- tasks.json is valid;
- selected Task exists;
- selected Task status is executable unless --allow-inactive is used;
- Task parent Epic exists unless --skip-plan-check is used;
- parent Epic is not inactive for active Task;
- parent Initiative is not archived for active Task;
- output path is writable when --write or --out is used.
```

Recommended future validation:

```text id="dzosqu"
- scope must not be empty for executable prompts;
- allowed_files must not be empty for executable prompts;
- acceptance_criteria must not be empty for executable prompts;
- generated prompt must match renderer output;
- prompt must include Stop Conditions;
- prompt must include Final Report requirements.
```

---

# 20. AI Consumption Rules

When Codex receives a Prompt Package, it must:

```text id="g2x4e1"
1. Treat the package as the active execution contract.
2. Stay within Scope.
3. Avoid Out of Scope items.
4. Edit only Allowed Files.
5. Use specified Verification Mode.
6. Report required final status.
7. Stop if required information is missing.
```

Codex must not:

```text id="2cj9hz"
- expand scope from Initiative or Epic;
- edit protected state files manually;
- change lifecycle state manually;
- invent acceptance criteria;
- invent allowed files;
- override Human Owner gates;
- implement system evolution without approved change;
- treat generated prompt as state.
```

# 21. Example Flow

## 21.1 Create Task

```bash id="khh4bk"
python scripts/taskctl.py task create \
  --epic EPIC-001 \
  --title "Implement protected files check" \
  --summary "Detect manual edits to protected project-control files." \
  --scope "Create scripts/check-protected-project-files.py" \
  --out-of-scope "No Codex plugin changes" \
  --allowed-file "scripts/check-protected-project-files.py" \
  --acceptance "Manual edits to generated files are detected" \
  --acceptance "Validation fails on outdated generated files" \
  --verification-mode standard
```

## 21.2 Select Current Task

```bash id="za7aac"
python scripts/taskctl.py current set TASK-001
```

## 21.3 Build Prompt

```bash id="n3g2og"
python scripts/taskctl.py prompt build --write
```

## 21.4 Execute

Codex reads:

```text id="c6m77b"
AI_PROJECT/generated/CODEX_PROMPT.md
```

and works only inside the generated contract.

## 21.5 Validate

```bash id="iqhbo5"
python scripts/taskctl.py validate
python scripts/taskctl.py check-generated
```

# 22. Failure Examples

## 22.1 No Current Task

Command:

```bash id="d5j719"
python scripts/taskctl.py prompt build
```

Expected error:

```text id="1t6rse"
NO_TASK_SELECTED
```

## 22.2 Inactive Task

Command:

```bash id="54v8p9"
python scripts/taskctl.py prompt build --task TASK-001
```

If `TASK-001` is `done`, `deferred` or `archived`, expected error:

```text id="xczof3"
TASK_IS_NOT_EXECUTABLE
```

Override for inspection only:

```bash id="bwh6f7"
python scripts/taskctl.py prompt build --task TASK-001 --allow-inactive
```

## 22.3 Missing Parent Epic

If Task references missing Epic, validation should fail:

```text id="zyrt6r"
VALIDATION_FAILED:
- tasks[0].epic_id references missing epic: EPIC-999
```

# 23. MVP Success Criteria

Prompt Package MVP is successful when:

```text id="f7crn3"
- prompt can be built for current Task;
- prompt can be built for explicit Task ID;
- prompt can be written to CODEX_PROMPT.md;
- prompt includes role, task, scope, out_of_scope, allowed_files and acceptance criteria;
- prompt includes execution rules;
- prompt does not include unrelated backlog;
- prompt build fails without selected Task;
- prompt build fails for inactive Task unless override is used;
- prompt build writes an audit event when output is written;
- Codex can execute using only generated prompt package.
```

# Summary

Prompt Package is the controlled execution contract between Project Control Gateway and Codex.

It is generated from Task state and optional Plan context.

```text id="1evw42"
Task state + Plan context
        ↓
taskctl.py prompt build
        ↓
CODEX_PROMPT.md
        ↓
Codex executes one bounded Task
```

The central rule:

```text id="sx6j2d"
Codex must not infer scope from the whole repository.
Codex must execute the generated Prompt Package.
Python owns prompt construction.
Human Owner owns approval and acceptance.
```
