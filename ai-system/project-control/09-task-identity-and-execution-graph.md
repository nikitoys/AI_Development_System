# Task Identity and Execution Graph

## Status

Draft

## Purpose

This document defines the target design for task identity, human-readable task references, backward compatibility with existing `TASK-XXX` records, explicit task dependencies and deterministic executable queue selection in Project Control Gateway.

This is a design document for `TASK-012` / `TIG-01`. It does not implement schema changes, CLI behavior, state migration or generated output changes.

## Scope

This document covers:

- the problem with global `TASK-XXX` identifiers;
- the target task identity model;
- epic-scoped human references such as `TIG-01`;
- epic key rules;
- backward compatibility for existing tasks;
- reference resolution behavior;
- dependency graph rules;
- ready vs executable task selection;
- deterministic executable queue sorting;
- future CLI surface;
- staged migration plan for `TIG-01` through `TIG-06`.

This document does not authorize automatic execution, automatic agent dispatch, automatic merge, automatic pull request creation, Human Owner acceptance, or migration of current project-control state.

## Problem Statement

The current task model uses global `TASK-XXX` identifiers. That is useful as a first machine-readable identifier, but it is not enough for large parallel work.

Current risks:

- global task numbers do not group related work inside an Epic;
- manually proposed task numbers can collide with already allocated tasks;
- task number can be mistaken for execution order;
- Epics may be planned and executed in parallel by default, but global numbering makes cross-epic ordering look implicit;
- generated Markdown is readable, but it must not become the source of truth for resolving task identity or execution order.

The key rule for the target model is:

```text
Task identity is explicit.
Task execution order is explicit.
Task number or reference does not imply execution order.
```

## Goals

- Provide stable machine identity for every Task.
- Provide readable human references for planning, prompts and review.
- Support epic-scoped task numbering.
- Preserve backward compatibility with existing `TASK-XXX` records.
- Support explicit cross-epic dependencies.
- Provide a deterministic executable queue.
- Keep generated files human-readable while preserving state files as source of truth.
- Keep future implementation bounded to later TIG tasks.

## Non-Goals

- No autonomous agent dispatch.
- No automatic merge, push or pull request creation.
- No L5 autonomy.
- No automatic task acceptance or Human Owner approval.
- No removal of legacy `TASK-XXX` compatibility in the MVP.
- No immediate state migration in `TIG-01`.
- No changes to `taskctl.py`, `planctl.py`, `docctl.py`, `evolutionctl.py` or current schemas in `TIG-01`.

## Source-Of-Truth Rule

Task identity and execution graph data must live in controlled project state, not in generated Markdown or chat history.

Target source of truth:

```text
AI_PROJECT/state/tasks.json
AI_PROJECT/state/plan.json
```

Readable generated files may display task references, aliases and dependency summaries, but they must remain disposable output rendered by the owning CLI.

## Target Identity Model

Each Task should eventually include these identity fields.

| Field | Type | Required | Meaning |
| --- | --- | --- | --- |
| `task.uid` | string | yes | Immutable machine identity for the Task. It never changes after creation. |
| `task.ref` | string | yes | Primary human-readable task reference, such as `TIG-01`. |
| `task.legacy_id` | string or null | compatibility | Existing global `TASK-XXX` identifier preserved during migration. |
| `task.aliases` | string array | yes | Additional references that resolve to the same Task, including old IDs where useful. |
| `task.epic_id` | string | yes | Parent Epic ID, such as `EPIC-004`. |
| `task.epic_key` | string | yes | Readable Epic key used as task reference prefix, such as `TIG`. |
| `task.local_seq` | integer | yes | Sequence number unique inside one Epic key. |
| `task.title` | string | yes | Human-readable Task title. |
| `task.status` | string | yes | Lifecycle status controlled by `taskctl.py`. |

Recommended meaning:

- `uid` is the immutable machine identity.
- `ref` is the primary human-readable reference.
- `legacy_id` preserves the old global `TASK-XXX` identifier.
- `aliases` contain old references and optional historical references.
- `epic_id` links the Task to `EPIC-XXX`.
- `epic_key` provides readable prefixes such as `TIG`, `ACP`, `DOC` or `CTX`.
- `local_seq` is unique inside one Epic key.

## UID

`task.uid` is the canonical immutable identity used by tools for durable references.

Recommended requirements:

- generated by the CLI at creation time;
- globally unique inside the project;
- stable across task rename, reparenting policy decisions and generated output changes;
- never reused after archival or deletion-like lifecycle operations;
- not meaningful to humans as execution order.

Suggested implementation shape for later tasks:

```json
{
  "uid": "task_01JZ0000000000000000000000"
}
```

The exact UID format is deferred to implementation, but it must be stable, globally unique and validation-friendly.

## Human Reference Format

The primary human reference format is:

```text
<EPIC_KEY>-<LOCAL_SEQ>
```

Examples:

```text
TIG-01
ACP-02
DOC-04
CTX-03
```

Reference rules:

- `EPIC_KEY` comes from the parent Epic.
- `LOCAL_SEQ` is unique within that Epic key.
- `LOCAL_SEQ` should be displayed with at least two digits for readability.
- `TIG-01` and `TIG-1` should not both be canonical references. The canonical display form should be normalized.
- References are globally unique while the Epic key exists.
- `uid` remains the true immutable identity even if a human reference policy changes later.

The reference must not imply execution order. It is a readable label, not a scheduler.

## Epic Key Model

Each Epic should eventually have an `epic.key` field.

| Field | Type | Required | Meaning |
| --- | --- | --- | --- |
| `epic.key` | string | yes after migration | Stable readable key used as the prefix for task references. |

Recommended format:

```text
^[A-Z][A-Z0-9]{1,7}$
```

Format rules:

- uppercase ASCII letters and digits only;
- starts with a letter;
- short enough for prompt and generated Markdown readability;
- unique across active and historical Epics unless an explicit archival reuse policy is later approved;
- not derived automatically from title when ambiguity exists.

Examples:

| Epic | Suggested Key | Notes |
| --- | --- | --- |
| Task Identity and Execution Graph | `TIG` | Current `EPIC-004` work. |
| Prompt Context Retrieval Layer | `CTX` | Existing context retrieval work. |
| Documentation Rails | `DOC` | Existing documentation-control work. |
| Agent Control Plane | `ACP` | Possible future agent-control work. |

Migration behavior for existing Epics:

- existing Epics keep `EPIC-XXX` IDs;
- a later TIG task assigns keys through an approved command;
- generated output may show both `EPIC-004` and `TIG` during transition;
- duplicate or missing keys must fail validation once the key field becomes required;
- if a key cannot be assigned safely, affected Tasks keep resolving by `legacy_id` until a Human Owner decision resolves the conflict.

## Backward Compatibility

Existing `TASK-XXX` identifiers must remain resolvable during the MVP.

Compatibility rules:

- existing `TASK-XXX` values remain accepted references;
- `TASK-XXX` may become `task.legacy_id`;
- `TASK-XXX` should also be included in `task.aliases`;
- CLI commands may accept `TASK-XXX` during transition;
- generated output may display both `task.ref` and `task.legacy_id` where useful;
- prompts may include both references to reduce ambiguity;
- old references must not be silently reassigned to another Task.

Example migrated Task:

```json
{
  "uid": "task_01JZ0000000000000000000012",
  "ref": "TIG-01",
  "legacy_id": "TASK-012",
  "aliases": ["TASK-012"],
  "epic_id": "EPIC-004",
  "epic_key": "TIG",
  "local_seq": 1,
  "title": "Document task identity and execution graph design",
  "status": "in_review"
}
```

## Reference Resolver

Project Control Gateway should provide one central Task reference resolver.

The resolver accepts:

- `uid`;
- `ref`;
- `legacy_id`;
- `alias`.

The resolver should not be duplicated across commands. `taskctl.py`, `codexctl.py`, `contextctl.py` and future review or execution commands should use the same resolution rules when they accept a task reference.

Recommended resolution flow:

1. Normalize the input reference according to the reference type where safe.
2. Search `uid`.
3. Search `ref`.
4. Search `legacy_id`.
5. Search `aliases`.
6. Return exactly one Task or fail with a stable error.

Validation must guarantee uniqueness across resolvable identifiers before command execution.

| Error | Meaning | Required behavior |
| --- | --- | --- |
| `TASK_REFERENCE_NOT_FOUND` | No Task matches the input. | Fail without mutation. |
| `TASK_REFERENCE_AMBIGUOUS` | More than one Task matches the input. | Fail and list matching task identities if safe. |
| `DUPLICATE_TASK_REF` | Two Tasks have the same `ref`. | Validation fails. |
| `DUPLICATE_TASK_ALIAS` | Alias appears on more than one Task or conflicts with another primary reference. | Validation fails. |
| `INVALID_TASK_REF_FORMAT` | Input cannot be a valid `uid`, `ref`, `legacy_id` or alias. | Fail before mutation. |
| `TASK_REF_CONFLICTS_WITH_LEGACY_ID` | A new `ref` collides with another Task's old `TASK-XXX`. | Validation fails. |

Resolver invariants:

- exact identity beats presentation convenience;
- ambiguous references never choose arbitrarily;
- generated Markdown is not used for resolution;
- aliases must be globally unique across all active and historical Tasks unless a future archival policy explicitly changes that rule.

## Execution Graph

Task execution order must be represented by explicit dependencies, not by IDs, refs or list order.

Recommended target field:

```json
{
  "depends_on": [
    {
      "ref": "TIG-03",
      "type": "hard"
    }
  ]
}
```

MVP dependency type:

| Type | Meaning |
| --- | --- |
| `hard` | The dependency must be satisfied before the dependent Task is executable. |

Future dependency types may include advisory or soft dependencies, but the MVP should implement only hard dependencies unless a later approved task expands the model.

Execution graph rules:

- dependencies may reference Tasks in the same Epic;
- dependencies may reference Tasks in another Epic;
- cross-epic ordering exists only when an explicit Task dependency exists;
- Epics may execute in parallel by default;
- Task ID, legacy ID, reference and local sequence do not imply execution order;
- dependency references must resolve through the central resolver;
- missing dependency references fail validation;
- dependency cycles fail validation.

Examples:

```text
TIG-05 depends on TIG-03
TIG-05 depends on TIG-04
```

This means `TIG-05` is not executable until the hard dependencies are satisfied. It does not mean every Task in `EPIC-004` waits for every Task in another Epic.

## Cycle Detection

The dependency graph must be acyclic for hard dependencies.

Validation must detect:

- direct cycles, such as `TIG-03 -> TIG-03`;
- two-task cycles, such as `TIG-03 -> TIG-04 -> TIG-03`;
- longer cycles across Epics;
- cycles introduced by aliases resolving to the same Task.

Cycle errors must report the cycle path using canonical task refs when available.

## Missing Dependency Detection

Validation must fail if a dependency reference cannot be resolved.

Examples:

```text
TIG-05 depends on TIG-99
TIG-05 depends on TASK-999
TIG-05 depends on unknown uid task_...
```

Missing dependencies must not be ignored, downgraded to warnings or treated as satisfied.

## Ready vs Executable

`ready` and `executable` are related but different.

`ready` means the Task is prepared for execution or selection. It has enough scope, allowed files, acceptance criteria and verification expectations.

`executable` means the Task can be started now.

MVP executable rule:

A Task is executable when:

- `status` is `ready`, or `planned` if the project policy allows planned execution;
- all hard dependencies are satisfied;
- parent Epic is `active` or `planned`;
- parent Epic is not `archived`, `deferred` or `done`;
- parent Initiative is not `archived`, `deferred` or `done`;
- no current `in_progress` Task conflicts with required locks or resources if a lock/resource model exists;
- project-control validation passes.

For the MVP, hard dependency satisfaction should require dependency Task status `done`.

Rationale:

- `approved` means review or approval passed, but the task may not be closed in the current lifecycle.
- `done` is the existing closed completion state.
- using `done` as the MVP satisfaction rule avoids treating review approval as completion before closure.

Open question: if the lifecycle later treats `approved` as sufficient for downstream work, the executable rule may be changed through a separate approved system evolution task.

## Deterministic Executable Queue

The executable queue is the deterministic sorted list of executable Tasks.

It is not a promise that every listed Task will be executed automatically.

Recommended sort order:

1. priority first;
2. parent Epic order;
3. `local_seq`;
4. `ref`;
5. `uid` as final tie breaker.

Priority rule:

- lower numeric priority sorts first unless the project later defines a different priority policy;
- missing priority should fail validation or be normalized by migration;
- ties must remain deterministic.

Epic order comes from controlled plan state. `local_seq` is used only inside the Epic key. It does not imply cross-epic ordering.

Example:

```text
1. TIG-01
2. CTX-04
3. TIG-02
```

This queue means these Tasks are currently executable and sorted deterministically. It does not mean all work will be auto-dispatched.

## Future CLI Surface

Later TIG tasks may add commands like these. `TIG-01` does not implement them.

```bash
python scripts/planctl.py epic create --key TIG --initiative INIT-001 --title "Task Identity and Execution Graph"
python scripts/taskctl.py task create --epic EPIC-004 --title "Document task identity and execution graph design"
python scripts/taskctl.py task resolve TIG-01
python scripts/taskctl.py task deps add TIG-05 --after TIG-03 --type hard
python scripts/taskctl.py task deps list TIG-05
python scripts/taskctl.py task executable
python scripts/taskctl.py task next
python scripts/taskctl.py task graph validate
python scripts/taskctl.py migrate identity
```

Command design requirements:

- every mutation must validate state before and after mutation;
- every mutation must append an audit event;
- generated Markdown must be rendered only by the owning CLI;
- commands must accept old `TASK-XXX` references during transition;
- commands must fail on ambiguous or duplicate references;
- no command may imply automatic dispatch, merge or acceptance.

## Migration Plan

Migration should be staged across the existing TIG task sequence.

| Phase | Task | Purpose | Expected result |
| --- | --- | --- | --- |
| 1 | `TIG-01` / `TASK-012` | Design only. | This source document exists and task moves to review. |
| 2 | `TIG-02` / `TASK-013` | Add Epic keys. | Plan model can store and validate unique `epic.key` values. |
| 3 | `TIG-03` / `TASK-014` | Add task identity fields. | Task state supports `uid`, `ref`, `local_seq`, `legacy_id` and `aliases`. |
| 4 | `TIG-04` / `TASK-015` | Add central resolver. | CLI can resolve `uid`, `ref`, `legacy_id` and `alias` consistently. |
| 5 | `TIG-05` / `TASK-016` | Add dependencies and executable queue. | Hard dependencies, graph validation and deterministic executable queue exist. |
| 6 | `TIG-06` / `TASK-017` | Migrate and update generated output. | Existing Tasks are migrated safely and generated Markdown displays new references. |

Safe migration principles:

- preserve all existing `TASK-XXX` identifiers;
- add new fields before changing command input behavior;
- validate duplicate refs and aliases before writing migrated state;
- render generated Markdown through the CLI after state changes;
- keep compatibility until Human Owner approves removal or hiding of old references;
- do not make generated Markdown the source of truth at any phase.

## Generated Output Expectations

Later generated task output may display both new and old references:

```text
TIG-01 (legacy: TASK-012) - Document task identity and execution graph design
```

Generated outputs should help humans read the plan, but state remains authoritative.

Generated files must include the standard warning header:

```text
GENERATED FILE. DO NOT EDIT MANUALLY.
```

## Invariants

- `uid` is immutable.
- `ref` is globally unique.
- `aliases` are globally unique.
- `legacy_id` remains resolvable during the MVP.
- `local_seq` is unique per Epic key.
- `epic.key` is unique.
- Task ID, legacy ID, ref and local sequence never imply execution order.
- Epics may execute in parallel by default.
- Cross-epic ordering exists only through explicit Task dependencies.
- Hard dependency references must resolve to exactly one Task.
- Hard dependency graph must be acyclic.
- Generated Markdown is not editable source of truth.
- Protected state changes only through the owning CLI.
- No automatic dispatch, merge, pull request creation or acceptance is authorized by this model.

## Validation Requirements For Future Tasks

Future implementation tasks should add validation for:

- missing `uid`;
- duplicate `uid`;
- missing `ref`;
- duplicate `ref`;
- invalid `ref` format;
- missing `epic.key` where required;
- duplicate `epic.key`;
- duplicate `local_seq` inside one Epic key;
- duplicate alias;
- alias conflicting with another Task's `uid`, `ref` or `legacy_id`;
- dependency reference not found;
- dependency reference ambiguous;
- dependency cycle;
- executable queue including a Task with unsatisfied hard dependencies;
- generated output drift after migration.

## Open Questions

- Should `approved` dependencies count as satisfied, or only `done` dependencies?
- Should task reparenting be forbidden, or allowed with alias history?
- Should locks and resources be part of the `TIG-05` MVP, or a later Epic?
- Should old `TASK-XXX` references remain visible forever, or only during transition?
- Should Epic keys be immutable after first Task creation?
- Should archived Epic keys be reserved forever to protect historical aliases?
